Decision Coordinator v2.9.2

---

1. System Goal

Guide a group from an open question to a clear decision as efficiently as possible.

---

2. Core System Model

System = EFSM (Extended Finite State Machine)

Statechart is:
- first-class artifact
- executable structure
- single source of truth

---

3. Architecture

User Input
→ LLM Interpretation
→ Structured Signals
→ Events
→ State Machine
→ Actions
→ LLM Output

---

4. Role of the LLM

LLM:
- interprets input
- produces signals
- generates language

LLM does NOT:
- change state
- trigger transitions
- decide outcomes

Rule:
LLM suggests, system decides

---

5. Signal Layer

LLM produces structured signals:

- preferences
- constraints
- uncertainty
- conflict signals
- objections

Signals influence:
- events
- guards
- actions

---

6. EFSM Definition

(state, event, context) → (next_state, actions)

Deterministic.
Centralized.

8. Participation Model

Participants may be:
- explicit
- implicit
- hybrid

Participation ≠ response

9. Initiator Model

Roles:
- decision authority

10. Participation Constraint (UPDATED)

min_participants defines the minimum number of participants required in the final solution set.

Default:

If not explicitly defined:
    min_participants = number of explicitly defined participants

This reflects a default expectation of full group coordination.

- threshold
- unanimity

Decision rule defines how agreement is evaluated,
NOT how many participants must be present.

26. Transition Function

transition(state, event, context)

Steps:

---

27. Event Coverage & Guard Semantics

Every relevant event must be explicitly handled in every state.

No implicit no-op allowed.

---

28. Decision Readiness (NEW CORE SECTION)

A decision can only be finalized if:

solution_found
AND participation_satisfied
AND NOT has_critical_unresolved_participants

---

29. Definition: participation_satisfied (UPDATED)

participation_satisfied = true if:

number_of_participants_in_solution ≥ min_participants

---

30. Definition: critical_unresolved_participants (NEW)

A participant is considered "critical unresolved" if:

their missing input could still affect:

- feasibility of the solution
- satisfaction of participation constraints
- outcome of the decision rule

Examples:
- could block consent
- could break majority
- required to reach min_participants
- has unresolved uncertainty affecting feasibility

---

31. Decision Principle (NEW)

The system does NOT wait for all participants.

The system waits only for participants whose input is still relevant.

Rule:

A decision may be finalized when:
- participation constraints are satisfied
AND
- no critical unresolved participants remain
