Decision Coordinator v2.9.3

---

1. System Goal

Guide a group from an open question to a clear decision as efficiently as possible.

---

2. Core System Model

System = EFSM (Extended Finite State Machine)

Statechart is:
- first-class artifact
- executable structure
- single source of truth

---

3. Architecture

User Input  
→ LLM Interpretation  
→ Structured Signals  
→ Events  
→ State Machine  
→ Actions  
→ LLM Output

---

4. Role of the LLM

LLM:
- interprets input
- produces signals
- generates language
- supports control via interpretation

LLM does NOT:
- execute transitions
- override guards
- finalize decisions

Rule:
LLM suggests, system decides

---

5. Signal Layer

Signals:
- preferences
- constraints
- uncertainty
- flexibility
- conflict
- objections

Influence:
- events
- guards
- actions
- quality evaluation

---

6. EFSM Definition

(state, event, context) → (next_state, actions)

Deterministic  
Centralized  

---

7. State Model

- clarifying
- collecting
- aggregating
- resolving
- validating
- avoiding
- deciding
- decided (terminal)
- infeasible (terminal)

---

8. Participation Model

Participants:
- explicit
- implicit
- hybrid

Participation ≠ response

---

9. Initiator Model

Roles:
- authority
- optional participant

---

10. Participation Constraint

min_participants defines minimum participants in solution.

Default:
min_participants = number of explicitly defined participants

---

11. Decision Rule

- initiator
- consent (default)
- majority
- threshold
- unanimity

---

12. Preference Model

Preference = stable  
Flexibility = modifier  

---

13. Aggregation Principle

Decision = aggregation of inputs

---

14. Reliability Model

Affects targeting, not outcome

---

15. Time Awareness

Object time vs process time

---

16. Uncertainty Handling

Clarify once → proceed

---

17. Constraint Validation

Validate before infeasible

---

18. Conflict Model

Loop:
aggregating → resolving → aggregating

---

19. Avoidance Model

Fallback:
→ consent

---

20. Infeasible Model

Valid terminal outcome

---

21. Terminal States

- decided
- infeasible

---

22. Events

- RESPONSE_RECEIVED
- AGGREGATION_COMPLETED
- VALIDATION_COMPLETED
- DECISION_CONFIRMED
- DECISION_REJECTED
- AVOIDANCE_DETECTED

---

23. Guards

- solution_found
- has_conflict
- needs_validation
- is_avoidance
- is_infeasible
- participation_satisfied
- decision_rule_satisfied
- confirmation_required
- has_critical_unresolved_participants

---

24. Confirmation Guard

confirmation_required = true if:
- rule requires it
- uncertainty exists
- fragile decision
- initiator needed
- OR decision_quality == low

Else false

---

25. Actions

- updateState
- askQuestion
- aggregateInputs
- handleConflict
- validateConstraint
- proposeDecision
- finalizeDecision

---

26. Transition Function

Steps:
1. match transitions
2. evaluate guards
3. pick first valid
4. execute actions
5. return next state

Fallback required

---

27. Event Coverage & Guard Semantics

All relevant events must be explicitly defined per state.

No implicit no-op.

Guards block transitions.

---

28. Decision Readiness

Decision allowed only if:

solution_found  
AND participation_satisfied  
AND NOT has_critical_unresolved_participants  

---

29. participation_satisfied

True if:

participants_in_solution ≥ min_participants

---

30. critical_unresolved_participants

Participant is critical if missing input can affect:
- feasibility
- participation constraint
- decision outcome

---

31. Decision Principle

Do not wait for everyone.  
Wait only for those who matter.  

---

32. DECIDING State

DECISION_CONFIRMED → DECIDED  
DECISION_REJECTED → AGGREGATING  

If confirmation_required = false:
→ system may internally emit DECISION_CONFIRMED  

---

33. Decision Quality Layer

After solution is found:

decision_quality ∈ {high, medium, low}

---

34. decision_quality Definition

Derived from:
- uncertainty
- alignment
- flexibility
- constraint rigidity
- conflict
- rule stability

---

35. Quality Classification

high → stable, aligned  
medium → moderate conflict  
low → fragile, uncertain  

---

36. Quality Semantics

NON-BLOCKING

Must NOT:
- block decisions
- affect guards

---

37. Quality Effects

- confirmation_required (low → true)
- explanation_level
- interaction style

---

38. Separation Principle

Correctness = guards  
Quality = evaluation  

Quality never overrides correctness  

---

39. Transition Table

[unchanged]

41. Migration

42. LLM Integration

==================================================
ADDENDUM — Decision Quality Formalization (Future Extension)
==================================================

decision_quality is currently heuristic.

Future versions should introduce:

decision_quality = f(context, signals)

---

Requirements:

- deterministic baseline (required)
- reproducible
- testable
- NON-BLOCKING
- does NOT affect transitions

---

Two-layer model:

1. Baseline Evaluation (required)

quality_score ∈ [0,1]

based on:
- uncertainty
- alignment
- flexibility
- conflict
- constraint rigidity

Mapped to:
- high
- medium
- low

---

2. LLM-Assisted Refinement (optional)

LLM may:
- refine interpretation
- generate explanations

LLM must NOT:
- override baseline
- affect transitions

---

Consistency Rule:

If baseline ≠ LLM:
→ baseline is authoritative

---

Principle:

Baseline ensures consistency  
LLM adds nuance  

---

Note:

Formalization is future evolution,
not required for current implementations.
