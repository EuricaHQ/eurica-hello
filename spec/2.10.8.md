37. AGGREGATION SEMANTICS (UPDATED)
==================================================

Aggregation must:

- attempt to combine compatible preferences across dimensions
- NOT collapse all compatible signals into a single flat value
- preserve complementary information

Example:

Input:
- Wednesday (day)
- Evening (time)

→ composite candidate:
  Wednesday evening

NOT:
→ Wednesday (alone)

---

37.1 Aggregation Completeness Rule

A candidate may be:

- compatible
- but incomplete

If required dimensions are missing:

→ candidate is incomplete

System must:

- NOT treat incomplete candidate as conflict
- NOT finalize immediately
- trigger clarification if material

---

37.2 False Completeness Prevention

The system must not assume completeness only because
available preferences are mutually compatible.

A candidate may still be under-specified.

---

37.3 Expected Structure Heuristic

The system may infer that additional dimensions are expected,
based on context and available preferences.

This is heuristic and non-blocking.

---

==================================================
37.4 COMPLETENESS ROUTING RULE (NEW)
==================================================

When evaluating transition from AGGREGATING:

IF solution_found == true
AND solution_complete == false:

→ The system MUST NOT transition to DECIDING

→ The system MUST remain in AGGREGATING
   OR transition to COLLECTING (clarification loop)

→ The system MUST trigger clarification behavior via:
   - Targeting Layer
   - Interaction Type Selection

→ interaction_type MUST be:
   - clarify_dimension (if dimension-related)
   OR
   - appropriate clarification type based on context

---

Clarifications:

- This rule is NOT a guard
- This rule does NOT block decision validity
- This rule ONLY affects transition routing

- Guard evaluation remains unchanged
- Guard priority remains unchanged

- Conflict priority remains dominant:

IF has_conflict == true:
→ transition MUST go to RESOLVING

---

Determinism Requirement:

This routing behavior MUST be explicitly implemented
in the transition logic.

Implicit interpretation is NOT allowed.

---

Testing Requirement:

Implementations MUST verify:

- AGGREGATING does NOT transition to DECIDING
  when solution_complete == false

- clarification is triggered instead

- behavior is consistent across runs

---

Rationale:

This rule ensures:

- no premature finalization of incomplete solutions
- deterministic routing behavior
- elimination of implementation ambiguity
