1. META

Purpose:
Define deterministic communication behavior for all interaction_types.

Scope:
Defines HOW messages are phrased.
Does NOT define system behavior.

---

General Constraints (MUST):

- concise
- single intent
- easy to answer
- unambiguous

---

Forbidden (MUST NOT):

- reveal system internals
- reference states, guards, transitions
- multiple independent questions
- bias user decisions (unless required)
- assume missing information

---

Structure Rules:

- one clear action or question
- minimal cognitive load
- no unnecessary explanation

---

Tone Constraints:

- neutral
- efficient
- no emotional language
- no personality projection

---

Determinism:

- must be testable via pattern checks

---

2. FRAMING

goal:
Clarify which dimensions belong to the decision.

constraints (MUST):

- single question
- neutral
- non-leading
- allow rejection

forbidden:

- implying necessity
- pre-selecting dimensions
- suggesting defaults

allowed:

- open clarification

fallback_template:
"Do we need to decide on this as well?"

examples:

GOOD:
- "Do we need to decide on the time as well?"
- "Should time be part of the decision?"

BAD:
- "We should also decide the time, right?"
- "Let's include time as well."

---

3. CLARIFY

goal:
Resolve ambiguity or missing information.

constraints (MUST):

- target one missing element
- specific
- answerable quickly

forbidden:

- vague requests
- multi-part questions
- repetition

allowed:

- direct, concrete questions

fallback_template:
"Can you specify this?"

examples:

GOOD:
- "Which day works for you?"
- "Do you prefer Wednesday or Thursday?"

BAD:
- "Can you clarify your availability?"
- "What do you think overall?"

---

4. CONFIRM

goal:
Confirm a proposed decision.

constraints (MUST):

- clearly state proposal
- ask for confirmation
- allow rejection

forbidden:

- forcing agreement
- implying inevitability

allowed:

- explicit confirmation

fallback_template:
"Is this okay?"

examples:

GOOD:
- "Shall we go with Wednesday evening?"
- "Is Wednesday evening okay for everyone?"

BAD:
- "We'll go with Wednesday evening."
- "This seems best, so let's do it."

---

5. INFORM

goal:
Communicate status or outcome.

constraints (MUST):

- no question
- concise
- factual

forbidden:

- hidden questions
- speculation

allowed:

- short statements

fallback_template:
"This is the current outcome."

examples:

GOOD:
- "Wednesday evening works for the group."
- "No conflicts detected."

BAD:
- "Wednesday evening works, right?"
- "I think this should work."

---

6. RESOLVE_CONFLICT

goal:
Resolve conflicting preferences.

constraints (MUST):

- clearly state conflict
- focus on resolution
- be direct

forbidden:

- blaming language
- vague conflict description

allowed:

- explicit comparison

fallback_template:
"These options conflict. Please adjust."

examples:

GOOD:
- "Wednesday and Thursday conflict. Which works for you?"
- "These options conflict. Can you adjust?"

BAD:
- "There seems to be an issue."
- "Maybe reconsider something?"

7. INTERACTION VALIDATION MODEL

---

7.1 Purpose

Ensure all generated messages comply with Interaction Spec constraints
before delivery. Deterministic, testable, and strictly separated from
core logic.

---

7.2 Validation Rules

Each generated message MUST pass ALL of the following:

Rule 1 — Single Intent
  Message contains exactly one question or one statement.
  Violation: multiple sentences with independent intents.

Rule 2 — No System Internals
  Message does not reference states, guards, transitions,
  or internal variable names.
  Violation: any internal term appears in output.

Rule 3 — Tone Compliance
  Message is neutral, efficient, and factual.
  Violation: emotional language, personality projection, speculation.

Rule 4 — Interaction Type Constraints
  Message satisfies all MUST constraints for its interaction_type.
  Message does not violate any forbidden rules for its interaction_type.

Rule 5 — No Logic Encoding
  Message does not encode decision logic, routing, or guard behavior.
  Violation: message implies system behavior or next steps beyond
  the immediate interaction.

---

7.3 Execution Order

Validation runs AFTER message generation, BEFORE delivery:

1. LLM generates message for interaction_type
2. Validate against rules 1–5
3. IF valid → deliver
4. IF invalid → regenerate (once)
5. IF still invalid → use fallback_template for interaction_type

---

7.4 Regeneration

On first validation failure:
- Pass original message + violation description back to LLM
- Request corrected output
- Apply same validation rules to corrected output

Maximum regeneration attempts: 1
Total attempts per interaction: 2 (initial + 1 regeneration)

---

7.5 Fallback

If validation fails after regeneration:
- Use the fallback_template defined for the interaction_type
- fallback_template is static, pre-validated, always compliant
- No further LLM calls

---

7.6 Separation Constraint

The validation model:
- operates ONLY on generated text
- does NOT inspect or modify system state
- does NOT influence transitions, guards, or routing
- does NOT read context beyond interaction_type and payload

Validation is a communication-layer concern ONLY.

---

7.7 Testability

Each validation rule MUST be implementable as a deterministic check.
Validation results MUST be reproducible across runs.

Tests MUST cover:
- valid message passes all rules
- each rule rejects its specific violation
- regeneration produces compliant output
- fallback is used after failed regeneration
- validation does not affect system state
