META SPEC v1.0.0 — Core Spec / Interaction Spec Separation

---

1. Purpose

Define the contract, separation, and consistency rules between:

- Core Spec
- Interaction Spec

Goal:

Ensure:
- deterministic system behavior
- controlled and testable communication
- strict separation of concerns

---

2. Artifact Definitions

Core Spec:

Defines:
- system behavior
- EFSM (states, events, transitions)
- guards and routing logic
- targeting (who)
- interaction_type (what)

Interaction Spec:

Defines:
- communication behavior
- phrasing constraints
- allowed / forbidden patterns
- examples

---

3. Responsibility Separation

Core Spec is responsible for:

- WHEN something happens
- WHAT happens
- WHO is targeted
- WHICH interaction_type is used

Interaction Spec is responsible for:

- HOW communication is phrased
- HOW constraints are applied
- HOW interaction intent is expressed

---

4. Interaction Contract

The interface between both layers is:

interaction_type

Core Spec:
→ emits interaction_type

Interaction Spec:
→ defines communication rules for that interaction_type

---

5. Strict Separation Rules

Interaction Spec MUST NOT:

- affect guards
- affect transitions
- affect routing
- affect targeting
- introduce system logic

Core Spec MUST NOT:

- define phrasing
- define tone or style
- include example messages
- encode communication behavior

---

6. Determinism Principle

System determinism is guaranteed exclusively by Core Spec.

Interaction Spec MUST NOT introduce non-deterministic behavior
that affects system outcomes.

---

7. Testability Principle

Core Spec tests:

- state transitions
- guard behavior
- routing logic
- completeness / conflict behavior

Interaction Spec tests:

- constraint compliance
- forbidden pattern absence
- structural message validation

---

8. Versioning Model

Core Spec and Interaction Spec:

- are versioned independently
- must remain compatible

Core Spec MUST reference:

→ Interaction Spec version

Example:

Decision Coordinator v2.11.1
→ uses Interaction Spec v1.0.0

---

9. Evolution Rules

Changes to Core Spec:

- must NOT implicitly change communication behavior

Changes to Interaction Spec:

- must NOT affect system logic

Breaking changes require:

- explicit compatibility update

---

10. Replacement Principle

Interaction Spec MUST be replaceable without:

- modifying Core Spec
- changing system behavior

Goal:

Allow:
- different LLMs
- different communication styles
- A/B testing

without affecting decision logic

---

11. Enforcement Principle

Violations of this separation are considered:

→ architectural errors

Examples:

❌ Core Spec defines message wording  
❌ Interaction Spec changes decision flow  
❌ LLM prompt encodes guard logic  

---

12. Summary

Core Spec:
→ deterministic decision system

Interaction Spec:
→ controlled communication layer

Meta Spec:
→ contract ensuring both remain cleanly separated
