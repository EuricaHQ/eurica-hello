CHANGELOG (v2.9.3 → v2.10)

Type: Core behavioral extension (Interaction / Targeting Layer)

Added:
- Interaction / Targeting Layer as core system component
- formal definition of target = f(context, signals, state)
- deterministic baseline targeting rules
- support for parallel targeting (multi-participant interaction)
- interaction_type model
- priority model for interaction sequencing

Clarified:
- separation of concerns:
  Guards → correctness
  Quality → confidence
  Targeting → efficiency
- uncertainty is not primary driver; decision impact is

Refined:
- role of LLM in interaction (refinement only, not control)
- 1:1 vs group communication rules

Unchanged:
- EFSM structure
- states, events, transitions
- decision readiness logic
- participation semantics
- decision quality layer

Principle introduced:
“Targeting optimizes how the system reaches a decision, not whether it can decide.”

Decision Coordinator v2.10

---

1. System Goal

Guide a group from an open question to a clear decision as efficiently as possible.

---

2. Core System Model

System = EFSM (Extended Finite State Machine)

Statechart is:
- first-class artifact
- executable
- single source of truth

---

3. Architecture

User Input  
→ LLM Interpretation  
→ Structured Signals  
→ Events  
→ State Machine  
→ Actions  
→ Targeting Layer  
→ LLM Output  

---

4. Role of the LLM

LLM:
- interprets input
- produces signals
- generates language
- refines interaction

LLM does NOT:
- execute transitions
- override guards
- select targets

Rule:
LLM suggests, system decides

---

5. Signal Layer

Signals:
- preferences
- constraints
- uncertainty (explicit)
- flexibility
- conflict
- objections

Used by:
- guards
- targeting
- quality evaluation

---

6. EFSM Definition

(state, event, context) → (next_state, actions)

Deterministic  
Centralized  

---

7. State Model

- clarifying
- collecting
- aggregating
- resolving
- validating
- avoiding
- deciding
- decided (terminal)
- infeasible (terminal)

---

8. Participation Model

Participants:
- explicit
- implicit
- hybrid

Participation ≠ response

---

9. Initiator Model

Roles:
- authority
- optional participant

---

10. Participation Constraint

min_participants defines minimum participants in solution.

Default:
min_participants = number of explicitly defined participants

---

11. Decision Rule

- initiator
- consent (default)
- majority
- threshold
- unanimity

---

12. Preference Model

Preference = stable  
Flexibility = modifier  

---

13. Aggregation Principle

Decision = aggregation of inputs

---

14. Reliability Model

Affects targeting, not outcome

---

15. Time Awareness

Object time vs process time

---

16. Uncertainty Model

Uncertainty may be:

1. explicit (LLM-derived)
2. implicit (missing / inconsistent data)
3. structural (decision impact of missing input)

Rule:
Structural uncertainty dominates when ambiguity exists.

---

17. Constraint Validation

Validate before infeasible

---

18. Conflict Model

Loop:
aggregating → resolving → aggregating

---

19. Avoidance Model

Fallback:
→ consent

---

20. Infeasible Model

Valid terminal outcome

---

21. Terminal States

- decided
- infeasible

---

22. Events

23. Guards

- solution_found
- has_conflict
- needs_validation
- is_avoidance
- is_infeasible

- confirmation_required
- has_critical_unresolved_participants

---

24. Confirmation Guard

confirmation_required = true if:
- decision rule requires it
- uncertainty exists
- fragile decision
- initiator needed
- OR decision_quality == low

---

25. Actions

- updateState
- askQuestion
- aggregateInputs
- handleConflict
- validateConstraint
- proposeDecision
- finalizeDecision

---

26. Transition Function

Deterministic selection via guards  
Fallback required  

---

27. Event Coverage & Guard Semantics

All relevant events must be explicitly handled  
No implicit no-op  

---

28. Decision Readiness

Decision allowed only if:

solution_found  
AND participation_satisfied  
AND NOT has_critical_unresolved_participants  

---

29. participation_satisfied

participants_in_solution ≥ min_participants

---

30. critical_unresolved_participants

Participants whose missing input could affect:
- feasibility
- participation
- decision outcome

---

31. Decision Principle

Do not wait for everyone  
Wait only for those who matter  

---

32. DECIDING State

DECISION_CONFIRMED → DECIDED  
DECISION_REJECTED → AGGREGATING  

If confirmation_required = false:
→ system emits DECISION_CONFIRMED internally  

---

33. Decision Quality Layer

decision_quality ∈ {high, medium, low}

Non-blocking

---

34. Quality Effects

- confirmation_required

- interaction behavior

---

35. Separation Principle

Correctness = guards

==================================================
36. INTERACTION / TARGETING LAYER (NEW CORE SECTION)
==================================================

---

36.1 Definition

Targeting determines:

WHO is interacted with  
WHAT interaction type is used  
WHEN interaction occurs  

---

36.2 Formal Model

target = f(context, signals, state)

Output:

target = {
  recipients: [participant_id | initiator | group],
  interaction_type,
  priority
}

---

36.3 Interaction Types

interaction_type ∈ {
  clarify,
  preference,
  constraint,
  resolve_conflict,
  confirm,
  inform
}

---

36.4 Targeting Principles

1. Prioritize decision impact over uncertainty  
2. Minimize communication overhead  
3. Prefer 1:1 interaction over group  
4. Parallelize when safe  

---

36.5 Targeting Priority Rules (Deterministic Baseline)

1. critical_unresolved_participants  

2. participants blocking participation_satisfied  

3. participants with missing constraints  

4. participants involved in conflict  

5. participants with highest uncertainty (fallback)  

---

36.6 Uncertainty Usage

Uncertainty is derived from:
- explicit signals
- implicit gaps
- structural impact

Rule:
Used only as fallback signal, not primary driver.

---

36.7 Parallel Targeting

Targeting may return multiple recipients.

Allowed when:
- interactions are independent
- no coordination conflict introduced

Typical cases:
- collecting preferences
- collecting availability

Not allowed when:
- resolving conflict
- confirming decision
- negotiation required

---

36.8 Priority Model

priority ∈ {high, medium, low}

high → blocks decision  
medium → improves quality  
low → optional  

---

36.9 1:1 vs Group Interaction

Default:
→ 1:1 interaction  

Group interaction only when:
- broadcasting results
- final decision communication
- group-level confirmation required  

---

36.10 LLM Interaction with Targeting

LLM may:
- refine phrasing
- enrich question context

LLM must NOT:
- override selected targets
- skip critical participants  

---

36.11 Targeting Outcome Principle

The system must prioritize interactions that reduce decision uncertainty
with minimal communication overhead.
