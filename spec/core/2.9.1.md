Decision Coordinator v2.9.1 (Fix: DECIDING State + Confirmation Semantics)

---

CHANGELOG (v2.9 → v2.9.1)

Type: Logical correction (bug fix, no conceptual change)

Issue:
- Deadlock in DECIDING state when confirmation_required = false
- No path to DECIDED without explicit confirmation

Fix:
- DECISION_CONFIRMED always leads to DECIDED
- confirmation_required now controls HOW the event is triggered (internal vs external), not IF it works

Added:
- Explicit rule: system may internally trigger DECISION_CONFIRMED if confirmation not required

Unchanged:
- State model
- Events
- Guards
- Core flow
- Event coverage principle

---

1. Corrected DECIDING State

STATE: DECIDING

RESPONSE_RECEIVED:
→ DECIDING (log / ignore / update minor signals)

AGGREGATION_COMPLETED:
→ DECIDING (irrelevant)

VALIDATION_COMPLETED:
→ DECIDING (irrelevant)

DECISION_CONFIRMED:
→ DECIDED

DECISION_REJECTED:
→ AGGREGATING

AVOIDANCE_DETECTED:
→ DECIDING (irrelevant)

---

2. Updated Confirmation Semantics

Definition:

confirmation_required determines the SOURCE of the DECISION_CONFIRMED event,
not the transition behavior.

---

Rule:

If confirmation_required = true:
- DECISION_CONFIRMED must be triggered externally
- via participant or initiator confirmation

If confirmation_required = false:
- system may internally emit DECISION_CONFIRMED
- no user confirmation is required

---

Implications:

- DECIDING always transitions to DECIDED via DECISION_CONFIRMED
- There is no implicit DECIDING → DECIDED transition
- Event-driven architecture remains intact
- No deadlock possible

---

3. Architectural Clarification

confirmation_required:
- does NOT block transition logic
- does NOT change state transitions
- ONLY determines whether confirmation must be collected from users

---

4. Consistency Check

This fix ensures:

- alignment with v2.8.1 intent:
  “Confirm only if needed, skip when safe”

- alignment with v2.9 principles:
  “All transitions are explicit and event-driven”

- elimination of deadlock scenario

---

5. Scope Confirmation

No other parts of the specification are affected.

- Transition table completeness unchanged
- Guards unchanged
- Events unchanged
- State model unchanged
- Behavior unchanged (only made consistent)
