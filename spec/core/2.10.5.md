18. CONFLICT MODEL (UPDATED)
==================================================

Conflict is a first-class control signal.

The system must not only detect conflicts,
but MUST actively resolve them.

---

Core Loop:

aggregating → resolving → aggregating

---

Principle:

Conflict is not passive.
Conflict MUST trigger active resolution behavior.

---

Prohibited State:

The system MUST NOT remain in AGGREGATING
while has_conflict == true.

24. GUARDS (UPDATED)
==================================================

Global Guard Priority Order:

1. has_conflict
2. needs_validation
3. solution_found
4. fallback

---

has_conflict:

Definition:
len(ctx.conflicts where status == "open") > 0

Semantics:

- BLOCKS transition to DECIDING
- ACTIVATES transition to RESOLVING
- HAS highest priority across all guards

---

needs_validation:

- evaluated only if no conflict exists

---

solution_found:

- evaluated only if:
  - no conflict
  - no validation needed

---

Principle:

Guards are evaluated in strict priority order.
Lower-priority guards must NOT override higher-priority conditions.

27. TRANSITION TABLE (UPDATED)
==================================================

AGGREGATING:

ON AGGREGATION_COMPLETED:

IF has_conflict:
→ RESOLVING

ELSE IF needs_validation:
→ VALIDATING

ELSE IF solution_found:
→ DECIDING

ELSE:
→ COLLECTING

---

Principle:

Conflict handling ALWAYS takes precedence.

==================================================
28. RESOLVING STATE (UPDATED)
==================================================

Entry Behavior (MANDATORY):

ON ENTER RESOLVING:

interaction_type = resolve_conflict

targeting:

- prioritize participants involved in conflicts
- MUST NOT default to group broadcast
- MUST focus on conflict resolution

---

Behavior:

- actively reduce conflict
- collect clarifications
- align preferences / constraints

---

Exit Condition:

When:

has_conflict == false

→ return to AGGREGATING
