Decision Coordinator v2.9

---

1. System Goal

Guide a group from an open question to a clear decision as efficiently as possible.

---

2. Core System Model

System = EFSM (Extended Finite State Machine)

Statechart is:
- first-class artifact
- executable structure
- single source of truth

---

3. Architecture

User Input
→ LLM Interpretation
→ Structured Signals
→ Events
→ State Machine
→ Actions
→ LLM Output

---

4. Role of the LLM

LLM:
- interprets input
- produces signals
- generates language

LLM does NOT:
- change state
- trigger transitions
- decide outcomes

Rule:
LLM suggests, system decides

---

5. Signal Layer

LLM produces structured signals:

- preferences
- constraints
- uncertainty
- conflict signals
- objections

Signals influence:
- events
- guards
- actions

---

6. EFSM Definition

(state, event, context) → (next_state, actions)

Deterministic.
Centralized.

---

7. State Model

States:
- clarifying
- collecting
- aggregating
- resolving
- validating
- avoiding
- deciding
- decided (terminal)
- infeasible (terminal)

---

8. Participation Model

Supports:
- explicit
- implicit
- hybrid

Participation ≠ response

---

9. Initiator Model

Roles:
- decision authority
- optional participant

---

10. Participation Constraint

min_participants defines valid solution size.

---

11. Decision Rule

- initiator
- consent (default)
- majority
- threshold
- unanimity

---

12. Preference Model

Preference ≠ flexibility

---

13. Aggregation Principle

Decision = aggregation

---

14. Reliability Model

Affects prioritization, not outcome

---

15. Time Awareness

Object time vs process time

---

16. Uncertainty Handling

Clarify once → proceed

---

17. Constraint Validation

Validate before infeasible

---

18. Conflict Model

Loop:
aggregating → resolving → aggregating

---

19. Avoidance Model

If no mechanism:
→ default = consent

---

20. Infeasible Model

Valid terminal outcome

---

21. Terminal States

- decided
- infeasible

No further transitions

---

22. Events (Core Set)

- RESPONSE_RECEIVED
- AGGREGATION_COMPLETED
- VALIDATION_COMPLETED
- DECISION_CONFIRMED
- DECISION_REJECTED
- AVOIDANCE_DETECTED

---

23. Guards

- solution_found
- has_conflict
- needs_validation
- is_avoidance
- is_infeasible
- participation_satisfied
- decision_rule_satisfied
- confirmation_required

---

24. Confirmation Guard

confirmation_required = true if:
- decision rule requires it
- decision is fragile
- critical uncertainty exists
- initiator approval required

Else false

---

25. Actions

- updateState
- askQuestion
- aggregateInputs
- handleConflict
- validateConstraint
- proposeDecision
- finalizeDecision

---

26. Transition Function

transition(state, event, context)

Steps:
1. find all rows
2. evaluate guards
3. pick first match
4. execute
5. return next state

Fallback row required

---

27. Event Coverage & Guard Semantics (NEW)

RULE:
Every relevant event must be explicitly handled in every state.

No implicit no-op allowed.

Guards control behavior.
Missing transitions are forbidden.

If event irrelevant:
→ explicit self-transition required

---

28. Complete Transition Table

CLARIFYING:

RESPONSE_RECEIVED:
- if clarification_complete → COLLECTING
- else → CLARIFYING

AGGREGATION_COMPLETED → CLARIFYING
VALIDATION_COMPLETED → CLARIFYING
DECISION_CONFIRMED → CLARIFYING
DECISION_REJECTED → CLARIFYING
AVOIDANCE_DETECTED → CLARIFYING

---

COLLECTING:

RESPONSE_RECEIVED → COLLECTING

AGGREGATION_COMPLETED:
- if participation_satisfied → AGGREGATING
- else → COLLECTING

VALIDATION_COMPLETED → COLLECTING
DECISION_CONFIRMED → COLLECTING
DECISION_REJECTED → COLLECTING

AVOIDANCE_DETECTED:
- if is_avoidance → AVOIDING
- else → COLLECTING

---

AGGREGATING:

RESPONSE_RECEIVED → AGGREGATING

AGGREGATION_COMPLETED:
- if is_infeasible → INFEASIBLE
- if needs_validation → VALIDATING
- if has_conflict → RESOLVING
- if is_avoidance → AVOIDING
- if solution_found → DECIDING
- else → COLLECTING

VALIDATION_COMPLETED → AGGREGATING

DECISION_CONFIRMED → AGGREGATING
DECISION_REJECTED → AGGREGATING

AVOIDANCE_DETECTED:
- if is_avoidance → AVOIDING
- else → AGGREGATING

---

RESOLVING:

RESPONSE_RECEIVED → RESOLVING
AGGREGATION_COMPLETED → AGGREGATING
VALIDATION_COMPLETED → RESOLVING
DECISION_CONFIRMED → RESOLVING
DECISION_REJECTED → RESOLVING

AVOIDANCE_DETECTED:
- if is_avoidance → AVOIDING
- else → RESOLVING

---

VALIDATING:

RESPONSE_RECEIVED → VALIDATING
AGGREGATION_COMPLETED → VALIDATING
VALIDATION_COMPLETED → AGGREGATING
DECISION_CONFIRMED → VALIDATING
DECISION_REJECTED → VALIDATING

AVOIDANCE_DETECTED:
- if is_avoidance → AVOIDING
- else → VALIDATING

---

AVOIDING:

RESPONSE_RECEIVED → AVOIDING
AGGREGATION_COMPLETED → AGGREGATING
VALIDATION_COMPLETED → AVOIDING
DECISION_CONFIRMED → AVOIDING
DECISION_REJECTED → AVOIDING
AVOIDANCE_DETECTED → AVOIDING

---

DECIDING:

RESPONSE_RECEIVED → DECIDING

AGGREGATION_COMPLETED → DECIDING
VALIDATION_COMPLETED → DECIDING

DECISION_CONFIRMED:
- if confirmation_required → DECIDED
- else → DECIDING

DECISION_REJECTED → AGGREGATING

AVOIDANCE_DETECTED → DECIDING

---

DECIDED:

ALL EVENTS → DECIDED

---

INFEASIBLE:

ALL EVENTS → INFEASIBLE

---

29. Testing Requirements

Each (state, event) must be tested.

Test must verify:
- event processed
- guard evaluated
- correct transition

Not allowed:
“event ignored”

---

30. Migration

Step 1: complete spec
Step 2: explicit transitions
Step 3: test alignment

31. LLM Relevance

Explicit transitions:
- improve traceability
- improve prompting
- enable reasoning

32. Statechart Requirement

Must include:
- all states
- all events
- all transitions
- guards
- fallback rows

No gaps allowed

---

33. Essence

No implicit behavior.

Everything:
- explicit
- guarded
- deterministic
- testable
