40. FRAMING FLOW MODEL (NEW)
==================================================

40.1 Definition

Framing Flow is the process by which the system establishes
which dimensions are relevant for the decision.

It operates:

- before completeness enforcement
- outside EFSM transitions
- via Targeting + Interaction Layer

---

40.2 Trigger Condition

Framing Flow MUST be triggered when:

- proposed_dimensions exists
AND
- len(proposed_dimensions) ≥ 2
AND
- expected_dimensions is None
AND
- no prior framing interaction has been executed

Framing Flow MAY additionally be triggered when:

- the system cannot construct a coherent composite candidate

---

40.3 Interaction Type

interaction_type = framing

Purpose:
- clarify decision scope (not decision content)

Properties:

- targeted at initiator
- non-blocking
- must not repeat excessively

---

40.4 Framing Interaction Behavior

System may ask:

- "Do we need to decide on [dimension]?"
- "Should we include [dimension] in the decision?"

System must:

- remain neutral
- avoid suggesting mandatory structure
- avoid leading the user

---

40.5 Initiator Response Mapping

Initiator response MUST be deterministically mapped to:

expected_dimensions ∈ {None, List[str]}

Examples:

- "Only day" → ["day"]
- "Day and time" → ["day", "time"]
- "Doesn't matter" → None

Rules:

- explicit inclusion → include dimension
- explicit exclusion → remove dimension
- neutral / ambiguous response → None

Mapping must be explicit and stored in context.

---

40.6 Authority Rule

Only initiator-defined or initiator-confirmed dimensions
may become expected_dimensions.

System must NOT:

- auto-promote proposed_dimensions
- enforce inferred dimensions

---

40.7 Non-Blocking Principle

Framing Flow must NOT:

- block aggregation
- block decision progression
- force user interaction

If no response is provided:

→ system continues with expected_dimensions = None

---

40.8 Frequency Constraint

Framing must occur:

- at most once per decision context
OR
- when new materially relevant dimensions emerge (future extension)

---

40.9 Separation Principle

Framing Flow is independent of:

- EFSM transitions
- guards
- conflict handling
- validation logic

It belongs exclusively to:

→ Targeting Layer
→ Interaction Type Selection

---

40.10 Integration with Completeness

Framing Flow affects completeness indirectly:

- expected_dimensions defined → completeness enforced

- expected_dimensions None → completeness ignored

---

40.11 Distinction from Clarification

Framing Flow ≠ clarify_dimension

Framing:
→ defines WHICH dimensions matter

clarify_dimension:
→ collects missing values for already defined dimensions

41. INITIATOR CLARIFICATION (UPDATED)
==================================================

When dimensional ambiguity affects decision framing:

System MAY:

- propose dimensions (via proposed_dimensions)
- trigger framing flow

Initiator may:

- confirm → becomes expected_dimensions
- reject → expected_dimensions = None
- modify → custom expected_dimensions

This clarification is:

- non-blocking
- separate from completeness enforcement
- part of decision framing

---

==================================================
42. TESTING REQUIREMENTS (UPDATED)
==================================================

Add framing tests:

- framing triggered when proposed_dimensions ≥ 2 and expected_dimensions None
- framing not triggered when expected_dimensions already defined
- framing occurs at most once
- framing does not block decision progression
- initiator response deterministically maps to expected_dimensions
- no implicit promotion from proposed_dimensions to expected_dimensions
- clarify_dimension only occurs after expected_dimensions defined

GLOBAL PRINCIPLES (UPDATED)

Add:

8. Framing Authority Principle

Decision dimensions must be explicitly defined or confirmed
by the initiator before they can influence completeness.

System inference alone is never sufficient.
