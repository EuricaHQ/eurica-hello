18. CONFLICT MODEL
==================================================

Conflict is a first-class control signal.

Core loop:
aggregating → resolving → aggregating

Conflict must be dimension-aware.

---

18.1 Complementarity Principle

Preferences that can be combined MUST NOT be treated as conflicting.

Example:
- Wednesday (day)
- Evening (time)

These are complementary, not conflicting.

---

18.2 Conflict Compatibility Rule

A conflict may only be created if:

- preferences refer to the SAME dimension
AND
- they are incompatible

If preferences refer to DIFFERENT dimensions:
→ they MUST NOT be treated as conflicting

If dimension is missing:
→ system must NOT assume conflict by default

---

18.3 Conflict Activation Principle

Conflict detection MUST lead to conflict resolution.

The system MUST NOT remain in AGGREGATING
while has_conflict == true.

24. GUARDS
==================================================

Global Guard Priority Order:

1. has_conflict
2. needs_validation
3. solution_found
4. fallback

---

has_conflict:

Definition:
len(ctx.conflicts where status == "open") > 0

Semantics:
- blocks transition to DECIDING
- activates transition to RESOLVING
- has highest priority

Conflicts must satisfy Conflict Compatibility Rule.

---

solution_found:

A solution exists based on aggregation.

---

solution_complete:

A solution is sufficiently complete for finalization if:

- no conflicts exist
- no critical participants are missing
- no critical dimensions are missing (if materially relevant and detectable)

Important:
solution_complete is NOT a hard blocking guard.
It is a behavioral routing refinement.

It may influence:
- clarification
- confirmation_required
- decision_quality
- finalization timing

25.7.1 Principle

36.10 INTERACTION TYPE SELECTION
==================================================

interaction_type = f(state, context, targeting_reason)

---

36.10.1 Baseline Mapping

IF state == clarifying:
    interaction_type = clarify

IF state == collecting:
    IF missing_preferences:
        interaction_type = preference
    ELSE IF missing_constraints:
        interaction_type = constraint
    ELSE:
        interaction_type = clarify

IF state == resolving:
    interaction_type = resolve_conflict

IF state == deciding:
    IF confirmation_required:
        interaction_type = confirm
    ELSE:
        interaction_type = inform

---

36.10.2 Targeting Reason Refinement

- missing_constraint → constraint
- missing_preference → preference
- conflict_participant → resolve_conflict
- uncertainty_only → clarify
- missing_dimension → clarify_dimension

---

36.10.3 Fallback

interaction_type = inform

---

36.10.4 Constraints

Must be:
- deterministic
- testable
- state-consistent

---

36.10.5 LLM Constraint

LLM must NOT override interaction_type

---

36.10.6 Separation

Targeting → WHO
Interaction Type → WHAT
LLM → HOW

36.10.7 Principle

System determines intent

37. AGGREGATION SEMANTICS (UPDATED)
==================================================

Aggregation must:

- attempt to combine compatible preferences across dimensions
- NOT collapse all compatible signals into a single flat value
- preserve complementary information

Example:

Input:
- Wednesday (day)
- Evening (time)

→ composite candidate:
  Wednesday evening

NOT:
→ Wednesday (alone)

---

37.1 Aggregation Completeness Rule

A candidate may be:

- compatible
- but incomplete

If required dimensions are missing:

→ candidate is incomplete

System must:

- NOT treat incomplete candidate as conflict
- NOT finalize immediately
- trigger clarification if material

---

37.2 False Completeness Prevention

The system must not assume completeness only because
available preferences are mutually compatible.

A candidate may still be under-specified.

Example:
- Wednesday
- Wednesday

may still be incomplete if time is materially required.

---

37.3 Expected Structure Heuristic

The system may infer that additional dimensions are expected,
based on context and available preferences.

This is heuristic and non-blocking.

---

==================================================
38. MISSING DIMENSION HANDLING
==================================================

The system may detect:

missing_dimension = true

IF:
- preferences exist
- multiple dimensions are present OR expected
- but no coherent sufficiently complete candidate can be formed

Behavior:

IF missing_dimension == true:

→ DO NOT finalize immediately
→ DO NOT create conflict
→ initiate clarification behavior

---

38.1 Materiality Rule

Missing dimensions should trigger clarification
ONLY IF they materially affect the decision.

The system must avoid over-questioning.

---

38.2 Dimension Uncertainty Rule

Dimensions are optional and uncertain signals.

If dimension is missing or inconsistently inferred,
the system must remain robust and should not over-rely on dimension logic.

---

==================================================
39. DIMENSION RELEVANCE
==================================================

A dimension is only required if:

- it is necessary to form a valid decision
OR
- it is defined or confirmed by the initiator

The system must NOT assume unnecessary dimensions.

Initiator may explicitly confirm or dismiss dimension relevance.

---

==================================================
40. INITIATOR CLARIFICATION
==================================================

When dimensional ambiguity affects decision framing:

System MAY ask initiator:

Example:
"Do you want to decide both day and time?"

This clarification is:

- non-blocking
- only used when ambiguity is material
- not required for simple decisions

Initiator may override dimension relevance.
