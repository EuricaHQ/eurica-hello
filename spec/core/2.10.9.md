38. MISSING DIMENSION HANDLING (UPDATED)
==================================================

The system may detect:

missing_dimension = true

IF:
- expected_dimensions is defined
AND
- at least one expected dimension is missing

Behavior:

IF missing_dimension == true:

→ DO NOT finalize immediately
→ DO NOT create conflict
→ initiate clarification behavior

---

38.1 Materiality Rule (UPDATED)

Missing dimensions trigger clarification ONLY IF:

- they are part of expected_dimensions

System MUST NOT infer materiality implicitly from signals alone.

---

38.2 Dimension Uncertainty Rule (UNCHANGED)

Dimensions are optional and uncertain signals.

System must remain robust without them.

==================================================
39. EXPECTED DIMENSIONS MODEL (NEW)
==================================================

39.1 Definition

expected_dimensions = optional list of dimension identifiers

Represents:
dimensions that are authoritatively considered materially relevant
for forming a complete decision.

---

39.2 Proposed Dimensions (NEW)

proposed_dimensions = optional list of inferred dimension identifiers

Represents:
system-inferred candidate dimensions based on:
- preferences
- language signals
- aggregation patterns

Important:
proposed_dimensions are NON-authoritative.

They must NOT:
- affect completeness routing
- block finalization
- trigger clarify_dimension directly

---

39.3 Sources of expected_dimensions

expected_dimensions may originate from:

A. Initiator-defined (authoritative)

- explicitly defined dimensions
- fully authoritative
- directly used for completeness evaluation

B. System-proposed + Initiator-confirmed

- system proposes candidate dimensions
- initiator confirms them
→ becomes authoritative expected_dimensions

---

39.4 Initiator Override Principle

The initiator may:

- define expected_dimensions
- confirm proposed_dimensions
- remove dimensions
- ignore dimensions

This decision is authoritative for completeness evaluation.

---

39.5 System Inference (Soft)

The system may:

- infer proposed_dimensions
- suggest them to initiator

The system must NOT:

- enforce them
- treat them as required
- use them in completeness routing

---

39.6 Completeness Evaluation Rule (UPDATED)

solution_complete may consider dimension completeness ONLY IF:

expected_dimensions is defined.

IF expected_dimensions is NOT defined:

→ completeness MUST NOT rely on dimension logic

→ system may proceed to DECIDING if other conditions are met

---

39.7 Clarification Trigger Rule (UPDATED)

clarify_dimension may only be triggered if:

- expected_dimensions is defined
AND
- at least one expected dimension is missing

---

39.8 Framing Clarification (NEW)

The system may ask initiator:

"Do we also need to decide on [dimension]?"

This is:

- framing clarification
- non-blocking
- independent of completeness enforcement

This MAY be triggered when:

- proposed_dimensions exist
- expected_dimensions is not yet defined
- ambiguity is material

---

39.9 Determinism Requirement

expected_dimensions must be:

- explicitly stored in context
- explicitly referenced in transition logic
- not inferred implicitly at guard time

All completeness behavior MUST depend only on:

- context
- explicit expected_dimensions

---

39.10 Non-Goals

This model must NOT:

- introduce fixed dimension taxonomy
- require dimensions for all decisions
- override conflict handling
- introduce new guards or states

40. INITIATOR CLARIFICATION (UPDATED)
==================================================

When dimensional ambiguity affects decision framing:

System MAY:

- propose dimensions (via proposed_dimensions)
- ask initiator for confirmation

Initiator may:

- confirm → becomes expected_dimensions
- reject → dimension ignored
- modify → custom expected_dimensions

This clarification is:

- non-blocking
- separate from completeness enforcement
- part of decision framing, not decision validation
